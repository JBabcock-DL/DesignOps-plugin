# Agent Handoff

This document is the standardized context transfer template for the DesignOps Plugin. When one skill invocation concludes and a subsequent skill needs to pick up where it left off, populate this file and pass it to the next invocation.

## Frontmatter Fields

| Field | Description |
|---|---|
| `active_file_key` | The Figma file key currently being worked on (e.g. `aBcDeFgHiJkL`). Find this in the Figma file URL: `figma.com/design/<file_key>/...` |
| `active_project_name` | The human-readable project name as it appears (or will appear) in the Figma team space (e.g. `Acme Mobile App`). |
| `last_skill_run` | The skill that was last executed (e.g. `new-project`, `create-design-system`). |
| `variable_slot_catalog_path` | Path to a variable slot catalog file, if used. Typically left empty — `/create-design-system` does not require it. |
| `token_css_path` | Relative path to the CSS token file written by `/create-design-system` (e.g. `src/styles/tokens.css`). Read by `/create-component` to wire the import into `globals.css`. Canonical WEB theme vars use **`--color-background*`** and **`--color-border*`** (Figma Theme groups `background/` and `border/`); shadcn **`--background`** aliases to `var(--color-background)`. Figma **`codeSyntax`**: ANDROID strings are **kebab-case** (e.g. `surface-container-high`); iOS strings are **dot paths** (e.g. `.Background.default`, `.Background.high`). |
| `open_items` | A list of things the next skill invocation should be aware of — unresolved questions, manual steps the designer still needs to take, or decisions deferred from the previous run. |

**Canvas vs variables:** Figma **variables** (collections, modes, `codeSyntax`) are the source of truth for code and dev mode. **Canvas frames** (doc headers on each page, style guide swatches, Token Overview copy, Thumbnail `Cover`, component pages populated by `/create-component`) are generated by `use_figma` per the skills (`/create-design-system` Steps 15a–18; `/sync-design-system` Steps 9b / 9d / 9e after a push to Figma; `/create-component` draws each component into its target `↳ {Page}`). The `[MCP] Token Manifest` frame and `↳ MCP Tokens` page were **removed** — do not reintroduce them. If a file was created before a skill update, re-run the relevant skill steps or the whole skill so canvas and variables stay aligned.

## How to Use This Handoff

1. **At the end of a skill run**, populate the frontmatter fields with the current state.
2. **Add any open items** that the next skill needs to know about — be specific (e.g. "Foundations file has not been renamed yet", "Style guide pages need redraw after manual variable edits", "`Cover` frame on Thumbnail needs manual layout fix").
3. **Pass this file to the next skill invocation** by including it in your Claude Code prompt:
   ```
   /create-design-system
   Context: see templates/agent-handoff.md
   ```
   (If your Claude Code workspace mounts this repo under a `plugin/` folder, use that path instead.)
4. **The receiving agent** should read this file first, incorporate the open items into its execution plan, and update the frontmatter before it concludes.

## Example — Populated Handoff

```yaml
---
active_file_key: "aBcDeFgHiJkLmNoP"
active_project_name: "Acme Mobile App"
last_skill_run: "new-project"
variable_slot_catalog_path: ""

  - "Foundations file created and page-scaffolded in Drafts — designer still needs to move it to Design-Systems/ in the team."
  - "Brand primary color: #E63946. All other tokens use defaults."
---
```
